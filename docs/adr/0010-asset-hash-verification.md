# ADR-0010: Asset SHA-256 Content Verification

**Date:** 2026-06-13
**Status:** Accepted
**Deciders:** @Anneardysa
**Builds on:** [ADR-0003](0003-multi-cdn-strategy-r2-primary.md), [ADR-0009](0009-cdn-download-resilience-layer.md)

## Problem Statement

ADR-0009 added download resilience and **size-only** validation, but explicitly deferred true
SHA-256 content verification because no server-side per-asset hash manifest existed. Without it,
a download that is corrupted in transit, truncated past the declared length, served stale by a
CDN edge, or tampered with would be extracted/installed/launched unnoticed. Skill Â§10 requires
"SHA-256 verification â€¦ after every completed download before file swap." Existing hashing
(`ModsPack.hash` whole-pack update detection; the exe anti-tamper hash) does not cover individual
asset downloads, and the app installer (`releases.json` / `UpdateInfo`) carried no hash at all.

## Decision

Publish a per-asset hash manifest and verify downloads against it on the client.

### Manifest contract â€” `Assets/asset_hashes.json`

```jsonc
{
  "version": 1,
  "algorithm": "SHA-256",
  "generatedAt": "2026-06-13T00:00:00Z",
  "assets": {
    "Assets/models/<hero>/<set>/model.zip": { "sha256": "<UPPERCASE-HEX>", "size": 12345678 },
    "Assets/Original.zip": { "sha256": "â€¦", "size": 987654321 }
  }
}
```

- Keys are repo-relative asset paths exactly as produced by `CdnConfig.ExtractAssetPath`
  (forward slashes); `sha256` is uppercase hex; `size` is bytes.
- **Split archives** are keyed by the **merged** `â€¦/model.zip` (the `.001` suffix dropped); the
  generator hashes the parts concatenated in order, matching the file the client assembles.
- Generated by `scripts/generate-asset-hashes.ps1` (run in the ModsPack repo before
  `sync-to-r2.ps1`).

### Scope verified

| Asset | Where verified | Expected hash source |
| --- | --- | --- |
| Hero set zip (single + merged split) | `HeroSetDownloaderService` (before extraction); single-file path inside `ResumableDownloadService` (per-CDN) | `asset_hashes.json` |
| `Original.zip` (base VPK) | `ResumableDownloadService` (per-CDN, via `OriginalVpkService`) | `asset_hashes.json` |
| App installer / portable | `UpdaterService` (after download, before apply/launch) | `releases.json` `installerSha256` / `portableSha256` fields |

### Enforcement semantics

- **Per-CDN (single file / Original.zip):** verification runs on the `.partial` before promotion.
  A mismatch deletes the partial, reports failure to the circuit breaker, and falls through to the
  **next CDN**. If every CDN fails verification â†’ `DownloadException(ErrorCodes.DL_HASH_MISMATCH)`.
- **Merged split + cache hits:** verified in `HeroSetDownloaderService` before extraction; a
  mismatch deletes the (corrupt or stale) cached file and throws `DL_HASH_MISMATCH`. This doubles
  as cache invalidation when the manifest hash changes for a path.
- **Installer/portable:** a mismatch deletes the temp file and aborts the update (never launched).
- **Graceful skip:** when the manifest is unavailable or an asset is absent, the existing
  size-only check applies â€” enabling staged rollout (publish the manifest at any time without a
  client release, and add assets incrementally).

### Components

- `Core/Services/Cdn/AssetHashEntry.cs`, `AssetHashVerifier.cs` (streamed SHA-256, uppercase hex),
  `AssetHashManifestService.cs` (singleton fetch+cache via the resilient
  `CdnFallbackService.DownloadStringWithFallbackAsync`, 10-min TTL, returns null on absence).
- `ResumableDownloadService.DownloadAsync` gains an optional `AssetHashEntry? expected`.
- `HeroSetDownloaderService` resolves the expected entry (resolver injectable for tests) and
  verifies single/split/cache paths.
- `UpdateInfo` gains `InstallerSha256` / `PortableSha256`, parsed from the R2 `releases.json`.

### Risk classification

`AssetHashVerifier` and `AssetHashManifestService` are `[AMT:OPUS]` (integrity gate). The
installer-manifest parsing is `[AMT:PRO]`.

## Considered Alternatives

- **Per-part hashes for split archives.** Would allow re-fetching a single bad part from another
  CDN, but bloats the manifest and the client logic. Rejected in favour of one merged-zip hash â€”
  parts already have per-part CDN fallback during download, and a post-merge mismatch re-downloads
  next run.
- **Signed manifest (embedded public key).** Stronger against a compromised origin, but heavier to
  operate. Deferred: the manifest is served over HTTPS from R2, and the threat model here is
  corruption/truncation/stale-edge/MITM-on-CDN, which a TLS-delivered hash manifest addresses.
- **Embed hashes in `heroes.json`.** Couples integrity to the hero catalog and omits non-set
  assets. Rejected for a dedicated, asset-type-agnostic manifest.

## Consequences

### Positive
- Corrupt/truncated/tampered downloads are rejected before extraction, install, or launch.
- A CDN serving a bad copy triggers fallback to the next CDN automatically.
- Manifest hash change invalidates stale client caches for that asset.
- Backward compatible: no manifest / missing entry â†’ size-only (no breakage during rollout).

### Negative
- Adds an extra manifest fetch (cached, resilient) and a streamed hash pass per verified download.
- Requires running `generate-asset-hashes.ps1` and publishing `releases.json` hashes as part of
  the release/sync workflow; an out-of-date manifest deletes a "good" cache and forces a
  re-download (self-healing, but extra bandwidth).

### Future work
- Optional manifest signing for origin-compromise resistance.
- Extend coverage to misc-mod archives if needed.
